# 职位内容改写方案（扒站 / 聚合来源）

> 目标：在**不虚构事实**的前提下，降低与来源站的逐字重复、提升可读性与 SEO 友好度；**公司主体信息**以可核验字段为准，谨慎改写。

---

## 一、适用范围与原则

| 维度 | 说明 |
|------|------|
| **适用** | 从外部抓取、导入的 JD 正文、任职要求、福利描述等**自由文本** |
| **原则上不改写（或仅规范化）** | 公司法定/展示名称、官网、对外电话/WhatsApp、**明确标注的薪资数字与币种**、**具体地址/城市**、职位类型（全职/兼职等）——与数据库字段一致 |
| **禁止** | 编造未在原文或结构化字段中出现的薪资、地点、班次、证书要求；将「招聘方」改成误导性主体 |
| **语言** | 对外展示为 **es-MX**；专有名词、品牌名保持原文写法 |

---

## 二、信息分层（改写前先归类）

1. **结构化事实层**（以 CMS / 导入列为准，改写文案时不得与之冲突）  
   - 公司名、城市、`salary_amount`、工作模式（现场/远程）、类别等  
2. **半结构化层**（可压缩、合并列表，不可改含义）  
   - 学历、经验年限、必备技能列表  
3. **叙述层**（主要改写对象）  
   - 公司介绍段落、岗位概述、重复套话、来源站营销话术  

---

## 三、输出格式规范（建议统一进库/HTML 前形态）

### 3.1 整体结构（推荐顺序）

```text
【岗位一句话】（1 行，≤120 字符）
【你将做什么】（无序列表，3～7 条，每条一行，动词开头）
【我们需要你具备】（无序列表，3～8 条）
【我们提供】（可选，无序列表，2～6 条）
【工作信息】（单行或小表格感：地点 / 模式 / 班次若原文有则写清）
```

### 3.2 Markdown / 富文本约定

| 规则 | 示例 |
|------|------|
| 列表统一用 `- `，层级最多 2 级 | 子项缩进 2 空格 |
| **不加** HTML `<h1>`；小节如需强调用 `**加粗**` 或独立一行短标题 | 避免与页面大标题重复 |
| 段落之间**空一行** | 便于渲染与移动端阅读 |
| 禁止整段全大写、禁止连续超过 3 个感叹号 | 降低「垃圾信息」观感 |
| 链接 | 仅保留白名单内链接（公司官网等）；不新增未核验外链 |

### 3.3 字数与密度（经验值）

| 区块 | 建议 |
|------|------|
| 岗位一句话 | 80～160 字 |
| 「你将做什么」+「需要具备」合计 | 300～900 字（过短易被判薄内容，过长易堆关键词） |
| 全文（不含重复粘贴的法律声明） | 控制在 **120～2000 字** 为主力区间 |

## 四、改写操作步骤（可给编辑 / 外包 / LLM 用）

### 步骤 A：预处理

1. 去除来源站导航、重复页脚、「分享至 xx」、HTML 垃圾标签。  
2. 将全角符号、异常空格、乱码（如 `Ã³`）做**字符修复**（与产品侧 `fixJobTextArtifacts` 等策略对齐）。  
3. 提取事实清单：薪资、地点、班次、证书——**单独列表**，改写正文时逐项核对。

### 步骤 B：重写叙述

1. **换骨架不换事实**：用不同句式、顺序重组「职责 / 要求 / 福利」。  
2. **同义替换有度**：避免为 SEO 堆砌城市名或职位名；自然出现 1～2 次主关键词即可。  
3. **去模板化**：删掉「世界五百强氛围」「挑战自我」等空洞句，改为具体动作或场景。  
4. **保留合规句**：若原文含平等机会、背景调查等法律声明，可**缩短保留**一句，勿自行杜撰法律条文。

### 步骤 C：质检（必做）

见 **第六节 QA 清单**；不通过不入库或不上线。

## 五、LLM 批量改写时的 Prompt 要点（摘要）

将以下内容作为系统/用户约束的一部分即可（按需粘贴）：

- 输入：原始 JD + 结构化字段（JSON 或表格）。  
- 输出：**仅** Markdown 正文，语言 es-MX。  
- 硬性约束：不得新增或修改薪资、城市、公司名；不得承诺原文未提及的福利。  
- 若原文与结构化字段冲突，**以结构化字段为准**，并在输出末尾用一行注释 `[NOTA: conflicto resuelto a favor de datos estructurados]`（该注释可不入库，仅人工审核用）。

## 六、QA 验收清单（每条内容上线前）

- [ ] 薪资、地点、公司名与 **数据库 / 导入列** 一致  
- [ ] 无「包过」「内推费」等违规或诈骗暗示  
- [ ] 无整段与来源站**连续相同**超过约 50 字（可用简单查重或抽检）  
- [ ] 列表、段落格式符合 **第三节**  
- [ ] 无乱码、无异常 HTML  
- [ ] 若职位已过期或下架策略变更，与 **noindex / 有效期** 产品规则无冲突  

---

## 七、与站内产品策略的衔接（当前实现）

- 职位详情 **`created_at` 起 90 天内** 仍可按规则参与索引与 JobPosting；超时配合 `is_active` 等逻辑会弱化 SEO 输出。  
- 改写**不会**重置 `created_at`；若希望「改写 = 重新计时」，需在**产品/数据层**单独定义字段（例如 `content_refreshed_at`），本方案默认**不改库表结构**。

---

## 八、版本与留痕（建议）

| 字段/动作 | 建议 |
|-----------|------|
| `description` / 正文版本 | 大改后记 `updated_at` 或版本号 |
| 来源 | 内部表或备注保留 `source_url`（不对外展示亦可），便于争议追溯 |
